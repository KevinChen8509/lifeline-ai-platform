## 1. 功能概述

用户创建订阅规则监控 IoT 设备数据，当数据满足预设条件时，平台通过 **Webhook HTTP POST** 将告警事件推送到用户预先配置的接收 URL，实现与第三方系统（运维平台、IM 机器人、工单系统、数据中台等）的实时集成。

### 1.1 核心价值

- **系统级集成**：通过 Webhook 将设备告警实时注入第三方业务系统
- **安全可信**：HMAC-SHA256 签名 + 时间戳防重放，接收方可验签确认来源合法
- **可靠投递**：指数退避重试 + 推送日志追溯，保证消息不丢失

### 1.2 与 v1.0 方案对比

| 维度 | v1.0（站内推送） | v2.0（Webhook 对接） |
|------|----------------|---------------------|
| 推送方式 | WebSocket + 站内信 + 邮件/短信 | Webhook HTTP POST |
| 接收方 | 平台内用户浏览器 | 第三方系统 HTTP 端点 |
| 鉴权 | Session/Token | HMAC-SHA256 签名验签 |
| 主要场景 | 运维人员实时查看告警 | 与外部系统自动化联动 |
| 前端重点 | 告警 Toast + 通知列表 | Webhook 配置 + 推送日志 |

## 3. 核心用户故事

### 3.1 Must（MVP 必须实现）

> 以下 5 个用户故事为核心 MVP 必须，US-WH-03 和 US-WH-04 降为 Should。

#### US-WH-01: 管理 Webhook 端点

> 作为系统集成人员，我想要配置一个 Webhook 接收地址（URL + 密钥），以便平台知道往哪里推送数据。

**验收标准**：
1. 能创建 Webhook 端点：填写名称、接收 URL、自定义 HTTP Headers
2. 创建时系统自动生成签名密钥（Secret），仅显示一次
3. 支持多端点管理（列表查看、编辑 URL/Headers、删除）
4. 提供端点连通性测试功能（平台发一条测试推送，验证可达性）
5. URL 必须为 HTTPS（生产环境）
6. **删除保护**：删除端点前检查是否有关联活跃订阅；有关联时弹出确认对话框，列出受影响的订阅名称和数量，需输入端点名称确认后才执行删除
7. **配额展示**：端点列表页顶部展示配额使用进度条（如"12/50"），精确到整数；配额 ≥80% 时进度条变为橙色预警，100% 时红色并禁用创建按钮；配额实时刷新（创建/删除操作后立即更新）

#### US-WH-02: 按设备实例创建数据订阅

> 作为运维人员，我想要订阅某台设备的温度数据点，当温度超过 80°C 时通过 Webhook 推送到我的系统。

**验收标准**：
1. 能选择具体设备实例（从设备列表/设备树中选择）
2. 能选择该设备下的具体数据点（如 temperature）
3. 能设置阈值条件（运算符 ≥/≤/==/!= + 阈值）
4. 能关联到一个已创建的 Webhook 端点
5. 触发后平台 POST JSON 数据到配置的 URL
6. 能在推送日志中查看该订阅的投递记录

#### US-WH-05: 推送日志与重试

> 作为系统集成人员，我想要查看每次 Webhook 推送的记录，包括成功/失败状态和响应详情。

**验收标准**：
1. 每次推送自动记录：时间、目标 URL、请求体、HTTP 状态码、响应体、耗时
2. 失败推送自动按指数退避重试（最多 5 次）
3. 推送日志支持按订阅/时间/状态筛选
4. 超过最大重试次数的推送标记为"投递失败"，可手动重试
5. 推送日志保留 30 天
6. 补偿推送（PENDING → 重发）时，推送日志标注 `source=COMPENSATION`，与正常推送区分
7. 降级场景下推送日志标注 `stormGuardDegraded=true` 或 `ruleMatchSkipped=true`；推送日志列表页支持按降级标记筛选；日志详情中降级标记以醒目黄色标签展示，hover 提示降级含义

#### US-WH-06: 推送签名验证

> 作为第三方系统开发者，我需要验证推送请求的来源合法性。

**验收标准**：
1. 每次推送携带 `X-Webhook-Signature`、`X-Webhook-Timestamp`、`X-Webhook-ID`、`X-Webhook-Event` 请求头
2. 签名算法 HMAC-SHA256，签名字符串 = `timestamp + "\n" + payload`
3. 平台提供签名验证示例代码（Python / Java / Node.js）
4. 时间戳超过 5 分钟的请求标记为过期（防重放）

#### US-WH-07: 订阅管理

> 作为运维人员，我想要查看、编辑、暂停、删除我的订阅。

**验收标准**：
1. 订阅列表支持按状态/设备/类型筛选
2. 支持单个启停/删除
3. 编辑时修改条件后立即生效
4. 订阅列表分页展示
5. 订阅状态变更后，自动化测试在 **P99 百分位**下，从 API 返回到规则引擎加载完成 ≤ 5 秒（连续 100 次无超时）；期间可能产生少量幽灵推送
6. **批量操作**：列表页支持多选订阅进行批量启用/停用/删除；批量操作前展示影响范围（N 条订阅，涉及 M 个设备）；批量操作结果展示成功/失败明细
7. **订阅状态转移**：支持幂等操作（重复启用/停用不报错）；订阅处于停用状态时推送任务暂停但不丢弃（队列缓存）；取消订阅时正在执行的推送完成当前请求后停止

5. 新密钥展示方式同创建时（一次性弹窗，仅展示一次）
6. 轮换操作记录审计日志（操作人、时间、端点 ID）
7. 宽限期内的推送使用新密钥签名

> 作为第三方系统开发者，我需要参考签名验证示例代码快速对接。

**验收标准**：
1. 平台提供 Python / Java / Node.js 三种语言的签名验证示例代码
2. 示例代码覆盖：签名计算、时间戳校验（5 分钟窗口）、防重放逻辑

4. 示例文档通过端点管理页「签名验证指南」入口访问

#### US-WH-10: 端点推送失败通知

> 作为订阅用户，当我的端点连续推送失败时，我希望收到站内通知。

**验收标准**：
1. 端点连续 5 次推送失败后，系统发送站内通知给端点创建者
2. 通知内容包含：端点名称、连续失败次数、最近失败时间
3. 通知支持「查看详情」（跳转推送日志）和「去修复」（跳转端点编辑）快捷操作
4. 端点恢复后自动发送恢复通知
5. 用户可配置通知偏好（通知开关、通知频率：每次/每天汇总）
6. 通知数据存入 `notification` 表，保留 30 天
7. 单条推送最终失败（DEAD 状态）后发送「推送最终失败」通知，包含 event_id 和失败原因
8. 补偿推送限流：补偿任务每次最多推送 10 条 PENDING 记录，避免对端点造成突发压力；10 条中失败的记录下次优先补偿（不重复推送已成功的）；同一设备在补偿队列中仅保留最新一条（去重）；补偿队列积压超过 100 条时在推送日志页展示积压告警提示

#### US-WH-12: 端点健康监控面板

> 作为运维人员，我需要查看端点的健康指标（成功率、延迟、熔断状态）。

**验收标准**：
1. 端点管理页每个端点展示健康指示灯（绿/黄/红/灰）
2. 点击健康指示灯展开详情面板：近 1h/24h 推送成功率、平均延迟、P95 延迟
3. 展示熔断器状态（CLOSED/OPEN/HALF_OPEN）
4. 健康指标通过 `GET /api/v1/webhook-endpoints/{id}/health` 获取（v1.0 REST API 轮询方式，前端 30s 自动刷新）
5. 指标缓存到 Redis（1 分钟 TTL），避免频繁聚合查询
6. 注：v1.1 将升级为 SSE 实时推送模式（见 US-WH-21），当前 API 接口设计预留 SSE 扩展能力

#### US-WH-21: SSE 实时健康监控面板（v1.1）

> 作为运维人员，我希望通过 SSE 实时接收端点健康状态变更推送，以便在端点异常时第一时间感知，而不依赖轮询间隔。

**验收标准**：
1. 健康监控面板建立 SSE 连接后，端点状态变更在 3 秒内推送到前端
2. SSE 断线后自动重连（指数退避 1s→2s→4s→8s→16s→30s 上限），重连超过 5 次降级为 10s REST 轮询
3. 重连期间通过 Last-Event-ID 机制补发未送达事件
4. SSE 连接数上限可配置（默认 1000），超限后新连接降级为轮询模式
5. 页面展示当前数据获取模式（SSE 已连接 / 轮询模式 / 连接中断）和断线重连次数
6. 与 v1.0 REST 轮询模式共存，无缝升级

#### US-WH-22: 规则冲突检测提示

> 作为订阅用户，我在创建或编辑订阅规则时，如果同设备已有规则存在冲突，系统应给出提示，帮助我避免重复或矛盾的配置。

**验收标准**：
1. 创建/编辑订阅规则时，系统实时检测该设备所有活跃订阅中是否存在冲突规则
2. **冲突定义**：同一设备 + 相同数据点 + 阈值范围重叠（如 rule1: temp>80, rule2: temp>70 → 重叠警告；rule1: temp>80, rule2: temp<30 → 无冲突）
3. 检测到冲突时以黄色警告标签展示冲突规则列表（含订阅名称、冲突数据点、阈值范围对比），**不阻止创建**
4. 相同条件的完全重复规则（同设备+同数据点+同运算符+同阈值）以红色"重复"标签展示，仍不阻止但强烈建议修改
5. 冲突检测覆盖跨订阅范围（同一用户对同一设备的所有活跃订阅）
6. 规则保存后冲突提示消失，但冲突信息记录在订阅详情中可供查看

## 4. 订阅模型定义

### 4.1 核心结构

```
WebhookEndpoint（推送端点）
├── URL: 接收推送的 HTTPS 地址
├── Secret: 签名密钥（AES-256-GCM 加密存储）
├── Headers: 自定义 HTTP 请求头
└── 状态: 启用/禁用

WebhookSubscription（订阅规则）
├── Target（观测目标）
│   ├── 类型: DEVICE / DEVICE_TYPE / DEVICE_GROUP
│   ├── 标识: 设备ID / 产品Key / 分组ID
│   └── 数据范围: DATA_POINT / DEVICE_STATUS / ALL_POINTS
│
├── Condition（触发条件）
│   ├── THRESHOLD: 阈值判断 [operator] + [value]
│   ├── STATUS_CHANGE: 设备离线/上线 [timeout]
│   └── ABSENCE: 数据停滞 [timeout]（V1.1）
│
├── Delivery（投递配置）
│   ├── 端点: WebhookEndpoint ID
│   ├── 重试策略: 指数退避 30s→60s→120s→240s→480s
│   ├── 最大重试次数: 5
│   └── 超时: 10 秒
│
└── Meta（元信息）
    ├── 创建者（用户ID）
    ├── 启用/停用状态
    └── 创建/更新时间
```

### 4.2 推送数据格式

### 事件类型枚举

| event_type | 含义 | 触发条件 |
|------------|------|---------|
| `alert.triggered` | 数据越限告警 | 数据点值满足阈值规则 |
| `device.offline` | 设备离线 | 设备离线超过设定时长 |
| `device.online` | 设备上线恢复 | 设备从离线恢复为在线 |
| `subscription.cancelled` | 订阅取消通知 | 设备被移除导致关联订阅停用 |

### 推送 Payload

```json
{
  "event_id": "evt_a1b2c3d4e5f6",
  "event_type": "alert.triggered",
  "timestamp": "2026-04-22T10:30:00.000Z",
  "subscription": {
    "id": 42,
    "name": "车间温度监控"
  },
  "device": {
    "id": 10086,
    "name": "温湿度传感器-A01",
    "device_key": "sensor_a01",
    "product_key": "temp_sensor_v2",
    "tags": { "location": "A栋1楼" }
  },
  "data_point": {
    "identifier": "temperature",
    "name": "温度",
    "value": 82.3,
    "unit": "°C",
    "reported_at": "2026-04-22T10:29:58.000Z"
  },
  "trigger_value": 82.3,
  "trigger_condition": ">80",
  "rule": {
    "id": 7,
    "type": "THRESHOLD",
    "description": "temperature > 80°C",
    "operator": "gt",
    "threshold": 80
  },
  "priority": "CRITICAL"
  
}
```

### 4.3 推送请求头

```http
POST {user_url} HTTP/1.1
Content-Type: application/json
X-Webhook-ID: evt_a1b2c3d4e5f6
X-Webhook-Event: alert.triggered
X-Webhook-Timestamp: 1745308200
X-Webhook-Signature: sha256=a3f5b8c2d1e4...
```

**签名算法**：
```
sign_str = timestamp + "\n" + json_payload
signature = Base64(HMAC-SHA256(secret, sign_str))
header_value = "sha256=" + signature
```

### 4.4 订阅粒度

| 粒度 | 描述 | MVP |
|------|------|-----|
| 按设备实例 | 订阅 device-001 的 temperature | **Must** |
| 按设备类型 | 订阅所有"温湿度传感器"的 temperature | **Must** |
| 按设备组 | 订阅"A栋1楼"所有设备 | Should |
| 按数据点类型 | 订阅所有设备的 temperature | Won't |

## 5. 非功能性需求

### 5.1 性能 SLA

| 指标 | Critical 级别 | General 级别 |
|------|-------------|-------------|
| 数据产生到发出 HTTP POST P95 | ≤ 2 秒 | ≤ 5 秒 |
| 数据产生到发出 HTTP POST P99 | ≤ 5 秒 | ≤ 10 秒 |
| 单次 HTTP POST 超时 | 10 秒 | 10 秒 |
| Webhook 并发推送能力 | 500 req/s | 500 req/s |

### 5.2 可靠性

| 指标 | 目标 |
|------|------|
| 消息投递保证 | at-least-once（接收方通过 event_id 去重） |
| 推送成功率 | ≥ 99.9%（含重试） |
| 推送服务 SLA | 99.9% |
| 重试策略 | 指数退避 2x，30s 起步，1800s 上限，最多 5 次 |
| 4xx 响应 | 不重试（接收方拒绝） |
| 5xx / 超时 | 自动重试 |

### 5.3 安全

| 措施 | 说明 |
|------|------|
| 签名验证 | HMAC-SHA256，防止伪造推送 |
| 防重放 | 时间戳 + 5 分钟窗口 |
| 密钥存储 | AES-256-GCM 加密存储，主密钥来自 Nacos/Vault |
| SSRF 防护 | 禁止内网地址（10.x / 172.16-31.x / 192.168.x / 127.x /private/ 内网地址 |
| HTTPS 强制 | 生产环境仅允许 HTTPS 端点 |

### 5.4 容量

| 维度 | V1 目标 | 扩展上限 |
|------|---------|---------|
| Webhook 端点数 | 500（每租户上限 50） | 5,000 |
| 活跃订阅数 | 5,000（每用户上限 100） | 100,000 |
| 推送日志日增量 | 50 万条 | 500 万条 |
| 并发 HTTP 推送 | 500 req/s（每租户上限 100 req/s） | 2,000 req/s |

**配额体验**：
- 创建端点/订阅达到上限时，返回明确错误提示："已达上限（23/50）"
- 端点管理页和订阅管理页展示配额使用情况
- 租户级推送限流：超出 100 req/s 的请求排队，不丢弃

### 5.5 数据保留

| 数据 | 保留策略 |
|------|---------|
| 推送日志（webhook_delivery_log） | 在线 30 天，归档 90 天 |
| Webhook 端点配置 | 随用户账号生命周期 |
| 订阅规则 | 随用户账号生命周期 |

### 5.6 运维需求

#### NFR-Ops-03: 灰度发布能力

> v1.1 Pipeline 2段式拆分上线时，需支持灰度发布，降低全量变更风险。详见 ADR-025。

- 四阶段灰度（5% → 20% → 50% → 100%），通过配置中心动态调整 endpointId 路由比例，无需重启服务
- 灰度期间旧 Pipeline 保持可用，支持一键回退（SLA < 3 分钟）
- 每阶段停留时间由运维根据健康指标决定；推荐指标：推送成功率 ≥ 99.5%、P95 延迟无劣化
- 灰度进度和各阶段健康指标可在运维面板查看
- 回滚触发条件：推送成功率 < 99% 或 P99 延迟 > 10s 时运维手动触发回滚

## 7. 迭代计划

### MVP（V1.0）—— "能配置、能推送、能追溯"

- Webhook 端点管理（CRUD + 连通性测试 + 密钥轮换）
- 按设备实例/设备类型订阅数据点阈值
- 设备离线/上线告警
- HMAC-SHA256 签名推送 + 签名验证示例文档
- 推送日志 + 指数退避重试
- 推送疲劳控制
- 订阅 CRUD + 启停
- 端点健康指示灯（绿/黄/红/灰）

### V1.1 —— "好用"

- 按设备组订阅
- 推送负载模板自定义（消息体格式）
- 端点健康监控面板 — SSE 实时推送健康状态（US-WH-21）
- 端点推送失败站内通知
- 设备详情页查看关联订阅
- 数据突变/停滞告警
- 批量推送（聚合多条告警为一次请求）
- 订阅规则模板（US-WH-23）
- Pipeline 2段式拆分（Ingestion+Matching 同进程消费，Dispatch 独立消费组），详见 ADR-024

**v1.1 灰度发布计划**（详见 ADR-025）：

| 阶段 | 流量比例 | 推荐停留时间 | 通过标准 | 回滚触发 |
|------|---------|-------------|---------|---------|
| 内部验证 | 5% | 1-2 天 | 核心路径 E2E 全通过 | 任何 P0 bug |
| 小规模 | 20% | 2-3 天 | 推送成功率 ≥ 99.9%，P99 延迟无劣化 | 成功率 < 99.5% |
| 半数 | 50% | 3-5 天 | 配额/补偿/降级功能全验证 | 成功率 < 99% |
| 全量 | 100% | 确认后锁定 | 全量指标回归 + 旧消费组下线 | 运维手动触发 |

**架构预留**：Endpoint Sharding（按端点分区并行推送）推迟到 v1.2，但当前数据模型已预留 shardKey 字段，v1.2 可无缝升级。

### V1.2 —— "智能化"

- 告警升级策略（连续失败时通知备选端点）
- 双向 Webhook（接收第三方回调确认）
- 推送效果追踪
- 告警关联分析
- Endpoint Sharding（按端点分区并行推送，32 分区）

#### 多规则匹配说明

当多规则同时匹配同一设备时：
- **推送策略**：每条匹配规则独立推送（不合并），保证接收方通过 event_id 去重的幂等性
- **推送顺序**：按规则优先级排序（Critical > Warning > Info），同优先级按规则创建时间
- **并发控制**：同一端点的推送请求受 Semaphore 控制，避免突发压力

## 9. 产品风险与缓解措施

> Party Mode 评审识别的产品风险，需在 v1.0 实施中关注。

| # | 风险 | 影响 | 缓解措施 | 涉及用户故事 |
|---|------|------|---------|------------|
| R01 | **缓存一致性延迟（幽灵推送）** | 订阅状态变更后 5s 内缓存未完全失效，可能产生 1-2 次幽灵推送 | Lua 脚本增加 `status=="ACTIVE"` 二次校验；推送日志标注 `cacheConsistentDelay=true`；用户文档说明 5s SLA | US-WH-07 |
| R02 | **推送最终失败无感知** | 单条推送重试耗尽（DEAD 状态）后用户无通知，依赖手动查看推送日志 | 新增「推送最终失败」站内通知，包含 event_id 和失败原因（US-WH-10 验收标准 7） | US-WH-10 |
| R03 | **降级状态不可见** | Redis 不可用时 StormGuard 降级到内存限流、Lua 跳过匹配，用户无法感知系统处于降级状态 | 推送日志增加降级标记字段（`stormGuardDegraded`、`ruleMatchSkipped`）；健康面板展示降级状态 | US-WH-05, US-WH-12 |
| R04 | **补偿推送突发压力** | 补偿任务批量重推 PENDING 记录时可能对端点造成突发压力 | 补偿任务限流：每次最多推送 10 条；推送日志标注 `source=COMPENSATION` | US-WH-05 |
| R05 | **配额耗尽静默失败** | 端点/订阅配额耗尽时用户创建失败，但可能不理解原因 | 返回明确错误信息 + 当前配额使用量；前端展示配额进度条 | US-WH-01, US-WH-02 |
| R06 | **v1.1 灰度发布回滚风险** | Pipeline 2段式拆分上线时灰度阶段可能触发未知问题，回滚不及时导致推送中断 | 4阶段灰度（5%→20%→50%→100%）+ 自动化健康指标监控 + 回滚 SLA < 3 分钟；详见 ADR-025 | NFR-Ops-03 |
| R07 | **规则冲突导致重复推送** | 用户对同一设备创建多条阈值重叠的规则，导致同一事件触发多次推送 | 创建/编辑时检测规则冲突并给出黄色警告（US-WH-22）；推送端通过 event_id 去重 | US-WH-22 |
