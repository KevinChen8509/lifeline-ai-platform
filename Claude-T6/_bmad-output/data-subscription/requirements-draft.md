# 数据订阅功能 - 需求草稿

> 本文档作为团队讨论的起点，各角色需在此基础上深化和完善。

## 1. 功能概述

用户可以订阅系统中的特定数据源（如传感器数据、设备状态、告警指标等），当订阅的数据发生变更或满足预设条件时，系统通过多种渠道（WebSocket 实时推送、邮件、短信）向用户发送通知。

## 2. 核心用户故事

### US-1: 浏览与发现数据源
作为用户，我希望浏览系统中所有可订阅的数据源，以便找到我感兴趣的数据。

### US-2: 创建订阅
作为用户，我希望对特定数据源创建订阅，并设置通知触发条件（如阈值、频率），以便只接收我关心的数据变更。

### US-3: 管理订阅
作为用户，我希望查看、编辑、暂停、删除我的订阅，以便灵活管理我的关注点。

### US-4: 接收实时通知
作为用户，当订阅的数据满足条件时，我希望通过 WebSocket 实时收到通知，以便第一时间了解数据变化。

### US-5: 配置通知偏好
作为用户，我希望设置通知的接收渠道（WebSocket/邮件/短信）和免打扰时段，以免被过多通知打扰。

### US-6: 查看通知历史
作为用户，我希望查看历史通知记录，以便回溯过去的数据变更事件。

## 3. 待团队讨论的问题

1. **数据源类型**: 系统中有哪些具体的数据源需要支持订阅？
2. **通知规则引擎**: 规则表达的复杂度需要到什么程度？简单的阈值？还是需要支持复合条件？
3. **推送性能目标**: 预期同时在线用户数？推送延迟要求？
4. **与现有告警模块的关系**: 数据订阅和现有告警功能如何区分和协作？
5. **消息可靠性**: 推送失败时的重试策略？离线用户的消息累积和补发？
6. **权限控制**: 不同角色对数据源的订阅权限如何定义？

## 4. 技术栈约束

- 后端: Java Spring Boot 3 + JPA
- 前端: Vue 3 + TypeScript + Element Plus
- 数据库: MySQL/PostgreSQL
- 缓存: Redis
- 消息队列: Apache Kafka（复用项目已有基础设施）
- 推送方式: Webhook HTTP POST（对接第三方系统）
- 容器化: Docker Compose
