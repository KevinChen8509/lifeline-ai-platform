# 城市生命线AI感知云平台 — 数据订阅推送规范

> **版本**: V2.0.0
> **日期**: 2026-04-28
> **依据**: 需求规格说明书 V1.0.1（用例 AIPC-06-001 / AIPC-06-002）
> **变更说明**: V2.0 采用自建平台推送标准（HMAC-SHA256 签名 + AES-256-GCM 加密），不兼容任何第三方平台协议

## 1. 订阅总体规范

### 1.1 订阅模式

| 维度 | 说明 |
|------|------|
| **推送协议** | HTTP/HTTPS POST（平台为客户端，客户服务器为接收端） |
| **订阅粒度** | 项目级 — 按 `projectId` 维度统一推送 |
| **地址模式** | 统一地址 / 独立地址可切换（参见 AIPC-06-001 业务规则） |
| **订阅状态** | `SUBSCRIBED`（已订阅，推送生效）/ `UNSUBSCRIBED`（未订阅，仅保存配置） |
| **服务周期** | 由项目 `serviceStart` ~ `serviceEnd` 自动控制订阅生效与失效 |

### 1.2 推送数据范围

| 数据类型 | 编码 | 说明 | 推送时机 |
|----------|------|------|----------|
| 设备信息 | `DEVICE_INFO` | 设备台账、型号、绑定关系变更 | 变更触发 |
| 运行状态 | `DEVICE_STATUS` | 在线/离线状态切换 | 状态变更时 |
| 实时监测数据 | `REALTIME_DATA` | 设备上报的指标数据（经校验清洗） | 按数据上报周期 |
| AI 诊断结果 | `AI_RESULT` | 溢流、淤堵、混接、满管诊断 | AI 分析完成时 |
| 报警信息 | `ALERT` | 超阈值报警生成/恢复/升级 | 报警状态变化时 |

### 1.3 订阅配置数据结构

```typescript
interface SubscriptionConfig {
  subscriptionId: string
  projectId: string              // 关联项目 ID（顶级隔离维度）
  name: string                   // 订阅名称
  pushUrl: string                // 客户接收推送的 URL
  secret: string                 // 平台生成的签名密钥（仅展示一次）
  encryptionMode: 'PLAIN' | 'AES' // PLAIN=明文，AES=AES-256-GCM 加密
  addressMode: 'UNIFIED' | 'INDEPENDENT'
  status: 'SUBSCRIBED' | 'UNSUBSCRIBED'
  dataTypes: DataType[]          // 订阅的数据类型列表
  createdAt: string
  updatedAt: string
}
```

## 2. 推送数据格式

### 2.1 推送信封格式（Envelope）

所有推送统一使用以下自建信封格式：

```json
{
  "version": "1.0",
  "id": "evt_20260428103000001",
  "timestamp": 1745700600000,
  "type": "ALERT",
  "signature": "hmac_sha256=v1,abcd1234...",
  "payload": { ... }
}
```

**字段说明：**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `version` | string | 是 | 信封格式版本号，便于将来格式演进 |
| `id` | string | 是 | 消息唯一 ID，格式 `evt_{timestamp}_{nonce}` |
| `timestamp` | long | 是 | 推送时间戳（毫秒） |
| `type` | string | 是 | 消息类型：`DEVICE_INFO` / `DEVICE_STATUS` / `REALTIME_DATA` / `AI_RESULT` / `ALERT` |
| `signature` | string | 是 | HMAC-SHA256 签名，格式 `hmac_sha256=v1,{hex}` |
| `payload` | object/string | 是 | 消息体。明文模式为 JSON 对象，加密模式为 AES-256-GCM 加密后的 Base64 字符串 |

### 2.2 签名校验规范

**算法：HMAC-SHA256**

```
1. 构造签名字符串: signingString = "{timestamp}\n{id}\n{payload的JSON字符串}"
2. 计算: hmacBytes = HMAC-SHA256(secret, signingString)
3. 编码: signature = "hmac_sha256=v1," + HexEncode(hmacBytes)
4. 对比: signature == 推送信封中的 signature 字段
```

**客户端校验步骤：**

1. 从信封取出 `timestamp`、`id`、`payload`
2. 用创建订阅时获得的 `secret` 密钥计算签名
3. 比较计算值与信封中 `signature` 字段
4. 建议校验 `timestamp` 与当前时间差不超过 5 分钟（防重放）

**为什么选择 HMAC-SHA256：**
- AWS SNS、Stripe Webhooks、GitHub Webhooks 等业界主流方案均采用此算法
- 提供密钥认证 + 消息完整性校验，安全性远高于 MD5 拼接签名
- 所有编程语言标准库均支持，客户端零额外依赖

### 2.3 安全模式加密

安全模式采用 **AES-256-GCM**（认证加密，AEAD）：

- **算法**: AES-256-GCM
- **密钥**: 平台生成 256 位随机密钥（创建订阅时展示一次）
- **IV**: 每条消息随机生成 12 字节 IV
- **输出格式**: `Base64(IV[12字节] + ciphertext + authTag[16字节])`
- **优势**: GCM 模式同时提供加密和完整性校验，无需单独 MAC

### 2.4 端点验证（首次绑定）

推送地址配置后，平台发送测试推送验证地址有效性：

```json
{
  "version": "1.0",
  "id": "evt_verify_test",
  "timestamp": 1745700600000,
  "type": "DEVICE_INFO",
  "signature": "hmac_sha256=v1,xxxx",
  "payload": {
    "verify": true,
    "message": "endpoint verification"
  }
}
```

客户端需：
1. 校验 `signature`，确认请求来自本平台
2. 响应 HTTP 200，响应体为 `{"verified": true}`

### 3.3 REALTIME_DATA — 实时监测数据

```json
{
  "projectId": "PRJ-20260427-001",
  "deviceId": "LC010100100001",
  "deviceType": "LC",
  "model": "LC01010010",
  "domain": "DRAINAGE",
  "timestamp": "2026-04-27T10:30:00+08:00",
```

### 3.4 AI_RESULT — AI 诊断结果

```json
{
  "projectId": "PRJ-20260427-001",
  "deviceId": "LC010100100001",
  "deviceType": "LC",
  "domain": "DRAINAGE",
  "timestamp": "2026-04-27T10:30:00+08:00",
  "diagnosisType": "OVERFLOW",
  "confidence": 0.92,
  "detail": {
    "code": "OVERFLOW_WARN",
    "message": "该点位检测到溢流风险，当前液位 2.35m，接近阈值 2.50m",
    "suggestion": "建议立即排查下游管道排水能力",
    "relatedMetrics": {
      "water_level": { "value": 2.35, "trend": "RISING", "rate": 0.15 }
    }
  }
}
```

**AI 诊断类型：**

| 编码 | 说明 | 触发条件 |
|------|------|----------|
| `OVERFLOW` | 溢流预警 | 液位持续接近或超过阈值 + AI 模型判定 |
| `SILTATION` | 淤堵检测 | 流量异常下降 + 液位异常上升 |
| `MIXED_CONNECTION` | 混接识别 | 水质指标异常（非雨水特征） |
| `FULL_PIPE` | 满管判定 | 液位达到管径 100% |

## 4. 排水设备数据点建模

### 4.1 设备类型定义

| type_code | 设备类型 | domain | 主指标 | 数据点数量 |
|-----------|----------|--------|--------|-----------|
| `LC` | 液位计 | DRAINAGE | water_level | 3+3(共有) |
| `LL` | 流量计 | DRAINAGE | instant_flow | 4+3(共有) |
| `YL` | 雨量计 | DRAINAGE | hourly_rainfall | 4+3(共有) |
| `SZ` | 水质一体机 | DRAINAGE | cod | 7+3(共有) |

> 设备编号 15 位规则：型号(8位) + 厂家码(2位) + 序号(5位)。
> 设备类型通过编号前 2 位（型号前缀）自动识别。

### 4.7 默认阈值配置

| 设备类型 | 指标 | 告警阈值 | 预警阈值 | 单位 |
|----------|------|----------|----------|------|
| LC | water_level | ≥ 2.50 | ≥ 2.00 | m |

## 5. 技术选型与实现架构

### 5.1 总体技术栈

| 层级 | 技术选型 | 版本 | 选型理由 |
|------|----------|------|----------|
| **运行时** | Java (JDK) | 18 | LTS 支持，Spring Boot 3.x 最低要求 17 |
| **应用框架** | Spring Boot | 3.2.5 | 生态成熟，自动配置，内嵌 Tomcat |
| **ORM** | Spring Data JPA + Hibernate | 6.4.4 | 领域驱动设计友好，减少 SQL 编写 |
| **数据库** | H2 (file-based) | 2.2.224 | 零运维嵌入式数据库，开发/演示阶段无需外部依赖 |
| **连接池** | HikariCP | 5.0.1 | 性能最优的 JDBC 连接池，Spring Boot 默认集成 |
| **JSON** | Jackson | 2.15.4 | Spring Boot 默认序列化框架 |
| **构建工具** | Maven | 3.9.6 | 标准化项目构建与依赖管理 |
| **前端** | Vue 3 + Element Plus (CDN) | 3.4.27 / 2.7.0 | 无需构建工具，快速原型验证 |
| **实时通信** | SSE (Server-Sent Events) | Spring SseEmitter | 单向推送场景最优解，比 WebSocket 轻量 |

### 5.2 推送引擎架构选型

#### 5.2.1 消息中间件：不引入 Kafka / MQTT

**决策：采用自建双层调度引擎，不引入外部消息中间件。**

| 对比维度 | Kafka | MQTT (EMQX/Mosquitto) | 自建调度引擎 (采用) |
|----------|-------|----------------------|-------------------|
| 运维复杂度 | 高（ZooKeeper/KRaft 集群） | 中（Broker 部署） | **零（纯内存 + DB）** |
| 数据可靠性 | 极高（副本 + 持久化） | 中 | **高（JPA 落库 + 重试调度）** |
| 吞吐量 | 百万级 TPS | 十万级 | **千级（满足当前场景）** |
| 适用场景 | 大规模事件流 + 流处理 | IoT 设备接入 | **平台内部推送编排** |
| 依赖成本 | JVM + 数 GB 内存 | 独立进程 | **无额外依赖** |

**选型理由：**
- 当前场景为平台内部 Webhook 推送编排，非海量设备数据接入，单机千级 TPS 足够
- 推送任务天然需要持久化到 DB 以支持重试和死信管理，Kafka 的追加日志模型在此场景反而冗余
- 目标环境为城市排水监测站，运维资源有限，减少外部中间件 = 降低故障面
- 未来若需横向扩展，PushTask 表可平滑迁移到 MySQL/PostgreSQL，调度层无感知

**中间件引入触发条件（Sprint 2 架构决策记录）：**

| 中间件 | 触发条件（任一满足即启动评估） | 当前值 | 状态 |
|--------|-------------------------------|--------|------|
| **Kafka** | 单日设备数据上报量超过 100 万条 | ~2.9 万/天（300 设备 × 96 条/天） | 远未达到 |
| | SSE 并发连接数超过 200 且出现明显延迟 | < 10 | 远未达到 |
| | 需要支持数据回放（replay）能力 | 暂无需求 | — |
| **MQTT** | 设备直连平台（当前通过网关 HTTP 转发） | 网关转发模式 | — |
| | 需要设备双向控制（如下发配置变更） | 暂无需求 | — |

> **评审周期**：每季度评估一次触发条件。下次评估时间：2026-Q3。

#### 5.2.2 双层调度架构

```
┌─────────────────────────────────────────────────────┐
│                  PushOrchestrator                     │
│            (统一入口：持久化 + 过滤 + 分发)              │
│   ┌─ 检查令牌桶限流 ─ 检查断路器 ─ 提交调度 ─┐        │
└──────────────────────┬──────────────────────────────┘
                       │
            ┌──────────▼──────────┐
            │   RetryScheduler    │
            │   (双层调度引擎)      │
            ├─────────────────────┤
            │ Layer 1: 内存链式调度 │
            │  ScheduledExecutor  │
            │  (4 线程, 快速重试)   │
            │  失败 → 计算退避延迟  │
            │       → schedule()  │
            │  超过3次 → 落库       │
            ├─────────────────────┤
            │ Layer 2: DB 扫表兜底 │
            │  @Scheduled(30s)    │
            │  JPQL: status IN    │
            │  (PENDING,          │
            │   RETRY_SCHEDULED,  │
            │   IN_MEMORY_RETRY)  │
            │  AND nextRetryAt ≤ NOW│
            │  ORDER BY priority DESC│
            │  LIMIT 50           │
            └─────────────────────┘
```

**设计原则：**
- **Layer 1**（内存）：处理即时推送和快速重试（1-3 次），延迟毫秒级
- **Layer 2**（DB 兜底）：定时扫表恢复所有待处理任务，包括 Layer 1 丢失的任务（进程重启后自动恢复）
- 两层互补：Layer 1 保证低延迟，Layer 2 保证不丢任务

### 5.3 核心组件技术实现

#### 5.3.1 断路器 (Circuit Breaker)

*Sprint 2 调参：failure-threshold 0.5→0.3（更早熔断保护下游），open-duration 60s→30s（更快探测恢复）。*

| 实现细节 | 值 | 说明 |
|----------|-----|------|
| 状态机 | `CLOSED → OPEN → HALF_OPEN → CLOSED` | AtomicReference 保证线程安全 |
| 滑动窗口 | `ConcurrentLinkedDeque<Boolean>` | 无锁，最近 10 次结果 |
| 失败率判定 | 50% | 窗口满后统计失败比例 |
| 开启时长 | 60 秒 | OPEN 状态持续 60 秒后转 HALF_OPEN |
| 半开探测 | 3 次成功 → CLOSED | 任意失败立即回到 OPEN |
| 注册中心 | `ConcurrentHashMap<String, CircuitBreaker>` | 按 endpointUrl 隔离 |

```
CLOSED (正常) ──失败率≥50%──→ OPEN (熔断)
  ↑                              │
  │ 3次探测成功                    │ 60秒后
  │                              ↓
  └──────── HALF_OPEN (半开探测)
                  │探测失败
                  ↓
               OPEN (重新熔断)
```

#### 5.3.2 限流器 (Rate Limiter)

| 实现细节 | 值 | 说明 |
|----------|-----|------|
| 算法 | 令牌桶 (Token Bucket) | 按 endpointUrl 隔离 |
| 令牌容量 | 100 个/分钟 | 每个端点独立计数 |
| 存储 | `ConcurrentHashMap<String, TokenBucket>` | AtomicInteger + AtomicLong |
| 报警冷却 | 300 秒 | 同一设备同一指标去重 |
| 超限行为 | 排队不丢弃 | 标记 RETRY_SCHEDULED，延迟 60 秒后重试 |

#### 5.3.3 退避策略 (Retry Policy)

```
delay = min(baseDelay × 2^attempt, maxDelay) × (1 ± jitter)
```

**Sprint 2 更新：按数据类型差异化重试策略（产品决策：告警必须送达，实时数据过时无意义）。**

| 参数 | 默认值 | 说明 |
|------|--------|------|
| baseDelay | 5000ms | 初始退避间隔（全局） |
| jitter | ±25% | 随机抖动，防止惊群效应 |

**按数据类型分级配置：**

| 数据类型 | maxRetries | maxDelay | 总窗口 | 理由 |
|----------|-----------|----------|--------|------|
| ALERT | 12 次 | 30min | ~4h | 告警必须送达，暴雨夜丢告警可能危及生命 |
| REALTIME_DATA | 3 次 | 5min | ~15min | 实时数据过时无意义 |
| DEVICE_STATUS | 8 次 | 10min | ~1h | 状态变化中等重要 |
| 其他（default） | 5 次 | 5min | ~3min | 通用兜底 |

**ALERT 退避时间表（无抖动时，baseDelay=5s）：**

| 重试次数 | 退避间隔 |
|----------|----------|
| 1 | 5s |
| 2 | 10s |
| 3 | 20s |
| 4 | 40s |
| 5 | 80s |
| ... | ... |
| 12 | 1280s (~21min, < 30min 上限) |

#### 5.3.4 推送任务状态机

```
                    ┌──────────┐
         submit()   │ PENDING  │
              ────→ └─────┬────┘
                           │ 立即执行
                           ↓
                    ┌──────────────┐
                    │IN_MEMORY_RETRY│ ← Layer 1 调度
                    └──────┬───────┘
                      成功  │     失败(>3次)
                      ↓     │        ↓
               ┌─────────┐  │  ┌──────────────┐
               │ SUCCESS │  │  │RETRY_SCHEDULED│ ← Layer 2 扫表
               └─────────┘  │  └──────┬───────┘
                            │         │ 重试达上限
                            │         ↓
                            │  ┌────────────┐
                            │  │ DEAD_LETTER │ ← 死信，需人工介入
                            │  └────────────┘
                            │
                     限流    │ 断路器OPEN
              ┌─────────────┐│┌──────────────┐
              │RETRY_SCHEDULED│││ CIRCUIT_OPEN │
              └──────────────┘│└──────────────┘
```

## 6. 推送可靠性保障

### 6.1 重推策略

采用标准指数退避 + 抖动（Exponential Backoff with Jitter）：

```
delay = min(baseDelay * 2^attempt + random(0, baseDelay), maxDelay)
```

| 参数 | 值 | 说明 |
|------|-----|------|
| baseDelay | 5秒 | 初始延迟 |
| maxDelay | 1小时 | 单次最大延迟 |
| maxRetries | 16 | 最大重试次数 |
| 响应超时 | 5秒 | 客户端需在5秒内返回 200 |

超过最大重试次数后标记为 `DEAD`，不再重试。

### 6.2 断路器保护

| 参数 | 值 | 说明 |
|------|-----|------|
| 失败率阈值 | 50% | 滑动窗口内 50% 推送失败则触发断路 |
| 滑动窗口 | 10次 | 最近 10 次推送统计 |
| 断路开启时长 | 60秒 | 断路开启后等待 60 秒进入半开 |
| 半开探测数 | 3次 | 半开状态允许 3 次探测请求 |

**断路器与重推联动：**
- 断路器 OPEN 时：不消耗重试次数，消息延迟 30 秒后重新入队
- 断路器 HALF_OPEN 时：允许单条探测，成功则 CLOSED，失败则重回 OPEN

### 6.3 风暴控制

| 参数 | 值 | 说明 |
|------|-----|------|
| 单端点最大推送频率 | 100次/分钟 | 超限排队，不丢弃 |
| 报警冷却时间 | 300秒 | 同一设备同一指标冷却期内不重复推送 |
| 离线检测冷却 | 3600秒 | 同一设备离线通知 1 小时内不重复 |

### 6.4 安全机制

| 模式 | 签名 | 加密 | 适用场景 |
|------|------|------|----------|
| 明文模式 | HMAC-SHA256 | 无 | 内网/测试环境 |
| 安全模式 | HMAC-SHA256 | AES-256-GCM | 公网/生产环境 |

## 7. 客户端接入指南

### 7.1 接入流程

```
1. 在平台配置推送地址
2. 平台发送测试推送 → 客户校验签名并返回 {"verified": true}
3. 配置消息推送规则（选择项目 + 数据类型 + 端点）
4. 订阅生效，平台开始 HTTP POST 推送
5. 客户校验 signature → 解密 payload（安全模式）→ 处理业务 → 返回 200
```

### 7.2 客户端响应要求

```http
HTTP/1.1 200 OK
Content-Type: application/json

{"received": true}
```

- 响应时间不得超过 5 秒
- HTTP 状态码必须为 200
- 不需要返回业务数据，仅需确认收到

### 7.3 签名校验示例

**Java:**

```java
import javax.crypto.Mac;
import javax.crypto.spec.SecretKeySpec;

public static boolean verifySignature(String payload, String timestamp,
                                       String id, String signature, String secret) {
    String signingString = timestamp + "\n" + id + "\n" + payload;
    try {
        Mac mac = Mac.getInstance("HmacSHA256");
        mac.init(new SecretKeySpec(secret.getBytes(), "HmacSHA256"));
        byte[] hmac = mac.doFinal(signingString.getBytes());
        String expected = "hmac_sha256=v1," + bytesToHex(hmac);
        return expected.equals(signature);
    } catch (Exception e) {
        return false;
    }
}
```

**Python:**

```python
import hmac, hashlib

def verify_signature(payload, timestamp, event_id, signature, secret):
    signing_string = f"{timestamp}\n{event_id}\n{payload}"
    expected = "hmac_sha256=v1," + hmac.new(
        secret.encode(), signing_string.encode(), hashlib.sha256
    ).hexdigest()
    return hmac.compare_digest(expected, signature)
```

**Node.js:**

```javascript
const crypto = require('crypto');

function verifySignature(payload, timestamp, id, signature, secret) {
  const signingString = `${timestamp}\n${id}\n${payload}`;
  const hmac = crypto.createHmac('sha256', secret);
  hmac.update(signingString);
  const expected = 'hmac_sha256=v1,' + hmac.digest('hex');
  return expected === signature;
}
```

### 8.4 V1.0 → V2.0 变更记录

| 变更项 | V1.0（已废弃） | V2.0（当前） | 原因 |
|--------|---------------|-------------|------|
| 信封格式 | 自建 `{version,id,timestamp,type,signature,payload}` | 自建平台，不需要兼容第三方 |
| 签名算法 | MD5 拼接签名 | HMAC-SHA256 | 安全性、业界标准 |
| 加密算法 | AES/CBC/PKCS7 | AES-256-GCM | 认证加密(AEAD)，无padding oracle风险 |
| 重推策略 | 固定16次退避表 | 指数退避+抖动 | 更简洁，自建平台无兼容约束 |
| 端点验证 | GET请求返回msg | 测试推送返回verified | 更自然的验证流程 |
| 数据点建模 | 无约束 | 类型安全+范围校验 | 消息质量保障 |
